# SDK initialization

## [VPBleCentralManager sharedBleManager]

* VPBleCentralManager: SDK main entrance
* sharedBleManager: Get the singleton object, the SDK interface exists in the form of a singleton

* Initialize VPBleCentralManage first, or you can initialize it in the main controller. In short, initialize it before using the Bluetooth function.

**Note⚠️, the internal data interface of the SDK does not support concurrent operations. For example, during the process of reading daily data, the on/off state of the operating device is not supported. In the event of concurrency, normal response to command operations cannot be guaranteed. The application layer needs to consider its own interactive logic processing in concurrent scenarios. **

**Note⚠️, due to system reasons, iOS allows multiple applications to connect to the same peripheral device at the same time. If there is H Band or SDK DEMO in your phone, please do not connect to the same device at the same time. It may trigger an infinite loop of reading data and cause an exception. **

The data generated by automatic measurement on the device side can be read to the APP side through the relevant interfaces of the SDK. To read the data generated by these automatic measurements, the SDK provides two methods. One is DB persistence inside the SDK. After the user calls **[Daily Data Reading Interface] and so on**, the persistent data object is obtained from the DB. The other is to read data directly through the interface and perform persistence operations outside the SDK.

It is recommended to use the SDK persistence method, which is less difficult to integrate and most customers choose this method.

SDK persistence method:

```swift
// VPPeripheralManage
VPBleCentralManage.sharedBleManager().peripheralManage = VPPeripheralManage.shareVPPeripheralManager()
```

Application layer self-persistence method:

```swift
// VPPeripheralAddManage
VPBleCentralManage.sharedBleManager().peripheralManage = VPPeripheralAddManage.shareVPPeripheralManager()
```

# Scan devices

### Prerequisites

When ensuring that the Bluetooth of the mobile phone system is turned on, if you enter the scanning device scanning interface and call the following code, there will be a callback of the device model after the device is scanned. A device may be scanned multiple times, so when displaying, it must be repeatedly filtered according to the device address attribute deviceAddress of the model VPPeripheralModel. User display can be performed based on the attribute value of the model.

 ### Class

  `VPBleCentralManager`

### API

```objective-c
- (void)veepooSDKStartScanDeviceAndReceiveScanningDevice:(ReceiveScanningDevice)scanningDevice;
```

### Parameters

| Parameters | Parameter type | Remarks |
| -------------- | ----------------------------------------------- | -------- |
| scanningDevice | void(^)(VPPeripheralModel *peripheralModel) | Device model |

### Custom scanning

When multiple SDK scenarios need to be integrated, internal scanning cannot be used.

The parameter `CBPeripheral` is obtained from the external `CBCentralManager` implementation of the scan delegation method. For relevant code, please refer to the `VPCustomScanManage` class in the Demo.

The scan will be re-executed internally, and the connection will be initiated only when the `CBPeripheral` with the same UUID is found.

```objective-c
///Used when self-implementing scanning for connected devices, suitable for integrating multiple SDK scenarios, the connection status is used: VPBleConnectStateChangeBlock
///Cannot be mixed with {@link veepooSDKConnectDevice:deviceConnectBlock:}
- (void)veepooSDKSelfScanConnectDevice:(CBPeripheral *)peripheral
                    deviceConnectBlock:(DeviceConnectBlock)connectBlock;
```

# Read device battery level

Special note⚠️: The dial transmission and OTA processes will consume a lot of power, and the power limit needs to be increased. When the power of the watch is very low, if you initiate dial transmission, the watch may shut down due to low power during the transmission process. After recharging, the dial will turn black. It is recommended to read the current battery level of the watch before each transmission. If the battery status is low, transmission is prohibited.

The OTA process is long, and it is recommended that the battery power be above 30% before the upgrade is allowed.

### Prerequisites

Device is connected

### Class

`VPPeripheralBaseManage`

Access method: **VPBleCentralManage.sharedBleManager().peripheralManage**`

### API

```objc
//Read the battery level of the device
- (void)veepooSDKReadDeviceBatteryInfo:(void(^)(BOOL isPercent, BOOL percenTypeIsLowBat, NSUInteger battery))deviceBatteryInfoBlock;

//Interface with charging status
- (void)veepooSDKReadDeviceBatteryAndChargeInfo:(void(^)(BOOL isPercent, VPDeviceChargeState chargeState, BOOL percenTypeIsLowBat, NSUInteger battery))deviceBatteryInfoBlock
```

### Parameters

| Parameters | Parameter type | Remarks |
| ------------------ | ---------- | --------------------------------------------------------------- |
| isPercent | BOOL | Indicates whether the battery level is a percentage |
| percenTypeIsLowBat | BOOL | Indicates whether the device is in a low power state when isPercent is true; |
| battery | NSUInteger | Battery power, if it is a percentage, the range is [0, 100]; otherwise, it indicates the number of cells of the battery, [0, 4] |

| Parameters | Parameter type | Remarks |
| ----------- | ------------------- | ---------------------------------- |
| chargeState | VPDeviceChargeState | Device charging state, enumeration types are detailed below |

```objective-c
//The charging status of the device
typedef NS_ENUM(NSUInteger, VPDeviceChargeState) {
    VPDeviceChargeStateNormal = 0, // The device is in normal use and not charging
    VPDeviceChargeStateCharging, // Charging
    VPDeviceChargeStateLowPressure, // low pressure state, has been abandoned
    VPDeviceChargeStateFull // Full state, unreliable
};
```

### Sample code

```swift
VPBleCentralManage.sharedBleManager().peripheralManage.veepooSDKReadDeviceBatteryInfo { isPercent, percenTypeIsLowBat, battery in
            let str = isPercent ? "electricity" : "voltage"
            let percentSign = isPercent ? "%" : ""
            weakSelf.vpDeviceBatteryLevelLabel.text = "Battery" + str + ":" + String(battery) + percentSign
        }
```

Example with charging status

```swift
veepooBleManager.peripheralManage.veepooSDKReadDeviceBatteryAndChargeInfo { [weak self]isPercent, chargeState, percenTypeIsLowBat, battery in
            let str = isPercent ? "electricity" : "voltage"
            var state = "normal use"
            if chargeState == .charging {
                state = "Charging"
            }
            if chargeState == .full {
                state = "full"
            }
            print("Charging status:\(state)")
            let percentSign = isPercent ? "%" : ""
            self?.vpDeviceBatteryLevelLabel.text = "Battery" + str + ":" + String(battery) + percentSign
        }
```

# Get step counting data

It refers to the step counting, distance and calories of the device. The data returned by this interface is real-time and is different from the number of steps in daily data. The number of steps in daily data is a summary of every 5 minutes, and there is a lag. If the application layer needs to obtain the number of steps on the device synchronously, it needs to call this interface at a fixed frequency to obtain the data.

### Prerequisites

*Device is connected

### Class

`VPPeripheralBaseManage`

Access method: **VPBleCentralManage.sharedBleManager().peripheralManage**

### API

SDK persistence method:

```objective-c
// 
+ (void)veepooSDKGetStepDataWithDate:(NSString *)queryDate andTableID:(NSString *)tableID changeUserStature:(NSUInteger)userStature result:(void(^)(NSDictionary *stepDict))resultBlock;
```

Application layer self-persistence method:

```objc
// If it is the data of the day, the device has been generating data, and the application layer needs to control the timing of reading. Data other than the current day is idempotent, and the result is the same every time it is read. The application layer can persist it by itself, and persistent data avoids repeated acquisition.
// Which day does dayNumber represent? 0 represents today, 1 represents yesterday, and 2 represents the day before yesterday. It cannot be greater than the value of the attribute saveDays in VPPeripheralModel.
- (void)veepooSDK_readStepDataWithDayNumber:(NSInteger)dayNumber result:(void(^)(NSDictionary *stepDict))readStepBlock;
```

### Callback parameters

| Parameters | Parameter type | Remarks |
| :----------: | :----------: | :-------------------------------------------------------------: |
| queryDate | NSString | The date to be queried, in the format of yyyy-MM-dd |
| tableID | NSString | Query the ID of the data and pass the MAC address of the device you want to obtain the data |
| userStature | NSUInteger | The user's height. Use this parameter to convert the number of steps into distance and calories. The height must be consistent with the personal information passed to the device. The default height of the device is 175 |
| resultBlock | NSDictionary | The returned format uses the specific name as the key and the specific value as the value. The format is as follows |

```objective-c
{
 Step = 5000;//Total number of steps per day
 Dis = 80.00; //The converted distance unit is km
 Cal = 1600.0; //The converted calorie unit is kcal (kilocalorie)
}
```

### Class

`VPPeripheralBaseManage`

```objective-c

/// - open: YES to start, NO to stop

                          result:(void(^)(VPDeviceGSRState state, NSInteger gsrValue))result;
```

### Parameters

VPDeviceGSRState

| State | Description |
| ----- | ----------- |
| `VPDeviceGSRStateStart` | Detection started on device |
| `VPDeviceGSRStateTesting` | Measurement in progress |
| `VPDeviceGSRStateComplete` | Measurement complete, `gsrValue` is valid |
| `VPDeviceGSRStateFail` | Detection failed |

```swift
    case .complete:
        print("GSR value: \(gsrValue)")
    case .fail:
        print("GSR detection failed")
    default:
        break
    }
}
```

## VPManualTestDataType enum

The `readManualTestDataWithTimestamp:dataType:result:` API accepts a `VPManualTestDataType` value to select which stored manual measurement data to retrieve. Known values (as of v1.1.8; earlier versions only supported `.bloodPressure`):

| Enum value | Description |
| ---------- | ----------- |
| `VPManualTestDataTypeBloodPressure` | Air-pump blood pressure measurements |
| `VPManualTestDataTypeHeartRate` | Heart rate measurements |
| `VPManualTestDataTypeTemperature` | Body temperature measurements |
| `VPManualTestDataTypeBloodOxygen` | Blood oxygen (SpO2) measurements |
| `VPManualTestDataTypeBloodComponent` | Blood component (cholesterol, uric acid, etc.) |
| `VPManualTestDataTypeBloodGlucose` | Blood glucose measurements |

Note: v1.1.8 extended `VPManualTestDataModel` to carry arrays for all six types above. Query each type separately and merge in the app layer. See live iOS wiki for the complete field list of each data model.
