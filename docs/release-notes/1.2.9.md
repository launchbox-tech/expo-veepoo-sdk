# 1.2.9

**Released:** Pending

## Summary

- **Native rejection normalization:** `AsyncFunction` failures from iOS/Android are parsed and mapped to stable **`VeepooError.code`** values (mutex, Session, capability, etc.) instead of often collapsing to **`UNKNOWN`**.
- **`VeepooError.nativeCode`:** Optional field when the bridge aliases or collapses a vendor string (e.g. **`START_FAILED`** → **`OPERATION_FAILED`**). See **ADR 0003** and **`CONTEXT.md`**.
- **`mapNativeRejection`** exported for advanced / test scenarios.

## Behaviour

- **`await` native methods** on **`VeepooSDK`** route errors through **`mapNativeRejection`** with a method-level fallback only when no native code is present.
- **Validators** still throw **`VeepooError`** directly and are **not** remapped.
- **No** message substring heuristics in this release.

## Upgrade impact

- **Additive:** `nativeCode?` on **`VeepooError`**; existing `switch (err.code)` branches gain more accurate codes for mutex and Session errors.
- Apps that stringify errors for logs may surface **`nativeCode`** for support.
