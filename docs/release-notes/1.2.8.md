# 1.2.8

**Released:** Pending

## Summary

- Realtime **manual vitals** APIs: **HRV**, **ECG**, **fatigue**, and **breathing** (`start*` / `stop*` pairs plus result events).
- **Global mutex:** Only one realtime health test at a time (includes HR, BP, SpO₂, temperature, stress, glucose, and the four vitals). Conflicting starts reject with **`REALTIME_TEST_IN_PROGRESS`**.
- New **`VeepooError`** codes consumed by these flows: **`CAPABILITY_UNSUPPORTED`**, **`REALTIME_TEST_IN_PROGRESS`**, **`DEVICE_NOT_READY`** (alongside existing session errors).
- Vendor parity matrix and example **Vitals lab** updated (#72).

## APIs & events

| Area | Methods | Events |
|------|---------|--------|
| HRV | `startHrvTest`, `stopHrvTest` | `hrvTestResult` |
| ECG | `startEcgTest(options?)`, `stopEcgTest` | `ecgTestResult` |
| Fatigue | `startFatigueTest`, `stopFatigueTest` | `fatigueTestResult` |
| Breathing | `startBreathingTest`, `stopBreathingTest` | `breathingTestResult` |

- **ECG:** Optional `EcgTestOptions`: `{ includeWaveform?: boolean }`. When `includeWaveform` is true, `ecgTestResult` may include a `waveform` number array (Band and platform dependent; can be large).
- **Types:** `HrvTestResult`, `EcgTestResult`, `EcgTestOptions`, `FatigueTestResult`, `BreathingTestResult` are exported from the package.

## Platform notes (traceability with #68–#71)

- **HRV:** Implemented on **Android**; **iOS** `startHrvTest` rejects with **`CAPABILITY_UNSUPPORTED`** (this bridge does not expose a realtime HRV manual test on the current Veepoo iOS bundle).
- **ECG, fatigue:** **iOS and Android** implement the same JS surface.
- **Breathing:** **iOS** implements the vendor breathing-rate test. **Android** initially rejected **`CAPABILITY_UNSUPPORTED`**; **`startDetectBreath`** is wired as of **[1.2.11](1.2.11.md)**.

Details and upstream links: [`docs/vendor-parity-matrix.md`](../vendor-parity-matrix.md) (Further notes section).

## Upgrade impact

- No breaking changes to existing method signatures; new symbols are additive.
- Host apps should expect **promise rejections** with stable `code` values when a second test is started, the Band does not support a modality, or the Session is not ready.
- If you call `startEcgTest({ includeWaveform: true })`, plan for **larger event payloads** and memory use.
